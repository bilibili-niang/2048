# 2048 游戏需求开发文档（Flutter 版）

## 1. 文档信息

- 文档名称：2048 游戏需求开发文档（Flutter）
- 文档版本：v2.1
- 更新日期：2026-05-11
- 适用项目目录：`flutter_2048/`
- 主要变更：
  - 新增首页（Home）作为统一入口
  - 新增继续游玩能力（恢复未完成对局）
  - 新增历史记录页（查看历史战绩）
  - 网格尺寸变更必须二次确认
  - 对局中不允许直接切换网格并改布局；变更网格只对“新开一局”生效
  - 新增随机方块生成概率规则（默认 2/4 概率）
  - 胜利目标从固定 2048 调整为与网格尺寸关联的动态目标

---

## 2. 项目目标与范围

### 2.1 项目目标

开发一款以 Flutter 为核心的 2048 游戏，重点提升“可持续游玩体验”和“操作安全性”，解决以下用户痛点：

1. 缺少首页导致入口混乱。
2. 用户退出后无法明确继续上一局。
3. 缺少历史战绩查看能力。
4. 修改网格尺寸会误操作影响当前对局。
5. 新方块生成数值单一（只生成 2）导致玩法失真。
6. 固定 2048 胜利条件无法适配不同网格难度。

### 2.2 本期范围（In Scope）

1. 首页（开始新游戏 / 继续游戏 / 历史记录 / 设置）。
2. 继续游玩（恢复最近未完成对局）。
3. 历史记录（按时间查看对局摘要）。
4. 网格尺寸调整流程重构：
   - 仅在“新开局流程”中修改。
   - 修改前必须二次确认。
   - 明确告知会开始新局，不影响当前保存局面（除非用户明确覆盖）。
5. 游戏核心规则修正：
   - 新方块按概率生成（2 和 4）。
   - 胜利目标按网格尺寸动态确定。

### 2.3 非本期范围（Out of Scope）

1. 联网排行榜。
2. 多设备云同步。
3. 用户账号系统。
4. 广告/内购系统。

## 3. 角色与用户故事

### 3.1 用户角色

- 普通玩家：希望快速开始、暂停后继续、回顾历史成绩。

### 3.2 核心用户故事

1. 作为玩家，我希望打开应用后先看到首页，而不是直接进入棋盘。
2. 作为玩家，我希望退出应用后，下次可以一键继续上一局。
3. 作为玩家，我希望查看最近玩过的记录，了解自己的进步。
4. 作为玩家，我希望改网格时有明确确认，避免误触清空当前节奏。
5. 作为玩家，我理解“改网格 = 新开一局”，所以不希望在对局中直接切尺寸影响当前布局。
6. 作为玩家，我希望新方块按经典概率出现（大部分是 2，少量是 4）。
7. 作为玩家，我希望不同网格有合理的目标方块，而不是永远固定 2048。

## 5. 功能需求（详细）

## 5.1 首页功能

### FR-HOME-001 首页基础布局

- 显示应用标题、最近最佳分、主操作按钮区。
- 按钮至少包含：
  - 新游戏
  - 继续游戏（有可恢复存档时可点击）
  - 历史记录
  - 设置

#### 验收标准

1. 首次安装打开应用，默认进入首页。
2. 无存档时“继续游戏”置灰并显示提示文案（如“暂无可继续对局”）。
3. 有存档时“继续游戏”可点击。

### FR-HOME-002 新游戏入口

- 点击“新游戏”后弹出网格尺寸选择面板（3x3 至 10x10）。
- 选择后必须触发二次确认弹窗（详见 5.4）。

### FR-HOME-003 继续游戏入口

- 点击“继续游戏”直接恢复最近一局未完成对局（含网格、分数、棋盘状态、时间戳）。
- 若恢复数据损坏或不存在，弹出错误提示并停留首页。

---

## 5.2 继续游玩（恢复对局）

### FR-CONT-001 自动保存策略

- 每次有效移动后自动保存当前对局快照。
- 快照至少包含：
  - `gridSize`
  - `grid`
  - `score`
  - `bestScore`
  - `hasWon`
  - `updatedAt`

### FR-CONT-002 恢复优先级

1. 优先恢复“最近一次未结束对局”。
2. 若最近对局已结束（失败或主动结束），则“继续游戏”按钮置灰。

### FR-CONT-003 恢复一致性

- 恢复后棋盘布局与退出前一致。
- 恢复后分数、网格尺寸、主题状态一致。

#### 验收标准

1. 在 4x4 对局移动 10 步后退出应用，再进入点击“继续游戏”，棋盘与分数一致。
2. 恢复后首次移动可正常继续，不出现重复生成方块或分数回滚。

---

## 5.3 历史记录

### FR-HIS-001 记录内容

每条历史记录至少包含：

1. 记录 ID（UUID）
2. 开始时间
3. 结束时间
4. 网格尺寸（如 4x4）
5. 最终分数
6. 最高方块（如 1024 / 2048）
7. 对局结果（进行中/失败/达成目标方块后继续/主动结束）
8. 总步数

### FR-HIS-002 记录时机

1. 新局开始时创建记录（状态：进行中）。
2. 游戏结束或用户主动结束时写入结束时间与最终状态。
3. 对于“进行中”记录，在关键进度变化时增量更新（如分数/步数）。

### FR-HIS-003 历史页交互

- 列表按 `结束时间或最近更新时间` 倒序。
- 支持空态展示（“暂无历史记录”）。
- 支持查看单条记录详情（可用弹窗或新页）。

### FR-HIS-004 数据保留策略

- 默认保留最近 100 条完整记录（可配置）。
- 超限时按最旧优先淘汰。

#### 验收标准

1. 完成一局后，历史页可看到正确记录项。
2. 连续对局后列表顺序正确，数据字段完整。
3. 空记录场景展示空态，不报错。

---

## 5.4 网格尺寸修改与二次确认

### FR-GRID-001 修改入口限制

- 对局进行中，**不提供直接修改网格尺寸入口**（或入口仅显示说明且不可执行）。
- 网格尺寸变更统一从首页“新游戏”流程进入。

### FR-GRID-002 二次确认弹窗（强制）

用户选择新网格后，必须出现确认弹窗，包含：

1. 目标网格（如“将开始 6x6 新游戏”）
2. 影响说明（“这将开启一局新游戏，不会延续当前对局布局”）
3. 操作按钮：取消 / 确认开始

### FR-GRID-003 行为约束

1. 点击“取消”：返回网格选择，不创建新局。
2. 点击“确认开始”：创建新局并进入对局页。
3. 当前进行中对局若已保存，不自动覆盖；仅当新局发生时更新“最近进行中对局”指针。

### FR-GRID-004 防误触与防抖

- 确认按钮点击后 500ms 内防重复触发。
- 连续点击不应创建多局重复记录。

#### 验收标准

1. 对局中无法直接改网格并刷新当前棋盘。
2. 选择网格后必须经过确认弹窗才能生效。
3. 取消操作不会改动当前任何对局数据。

## 5.5 对局状态与规则补充

### FR-GAME-001 对局状态机

定义状态：

1. `idle`：未开始（首页）
2. `running`：进行中
3. `won`：达成当前网格目标方块（可继续）
4. `over`：结束
5. `archived`：已归档历史

状态转换约束：

1. `idle -> running`：新游戏或继续游戏
2. `running -> won`：首次达到当前网格目标方块
3. `running/won -> over`：无法移动或用户主动结束
4. `over -> archived`：写入历史完成

### FR-GAME-002 返回首页行为

- 对局页返回首页前先触发一次保存。
- 首页继续游戏按钮状态实时更新。

## 5.6 核心数值规则（新增）

### FR-RULE-001 新方块生成概率

- 每次有效移动后，在随机空位生成 1 个新方块。
- 新方块数值采用默认概率：
  - `2`：90%
  - `4`：10%
- 概率允许在代码层集中配置，便于后续活动模式扩展。

#### 验收标准

1. 连续记录 1000 次生成结果，`2` 出现比例应在 85%~95% 区间。
2. `4` 必须可稳定出现，不允许只生成 `2`。

### FR-RULE-002 胜利目标动态化

- 胜利目标不再固定为 2048，而是根据当前网格尺寸 `gridSize` 决定 `targetTile`。
- 默认目标映射如下：

| 网格 | 目标方块 |
| :--- | :--- |
| 3x3 | 256 |
| 4x4 | 2048 |
| 5x5 | 4096 |
| 6x6 | 8192 |
| 7x7 | 16384 |
| 8x8 | 32768 |
| 9x9 | 65536 |
| 10x10 | 131072 |

- 当任意方块值 `>= targetTile` 时判定达成胜利（进入 `won` 状态），用户可选择继续游玩。

### FR-RULE-003 UI 目标展示

- 对局页需展示当前目标（如“目标：4096”）。
- 胜利弹窗文案需带入当前目标（如“你达成了 4096！”）。
- 历史记录详情需展示“目标方块”和“是否达成目标”。

#### 验收标准

1. 切换不同网格开新局时，目标值同步更新。
2. 在 5x5 对局合成 4096 时触发胜利，在 2048 时不触发胜利。
3. 在 3x3 对局合成 256 时触发胜利。

## 6. 数据结构与存储设计

## 6.1 本地存储 Key 规划（SharedPreferences）

1. `active_game_state`：当前可继续对局快照（JSON）
2. `active_game_exists`：是否存在可继续对局（bool）
3. `history_records`：历史记录列表（JSON Array）
4. `best_score_by_grid`：不同网格最高分映射（JSON）
5. `theme_mode`：主题模式（string）
6. `vibration_enabled`：震动开关（bool）
7. `rule_profile_version`：核心规则版本号（用于兼容未来规则调整）

## 6.2 继续对局结构（示例）

## 7. 交互与文案规范

## 7.1 关键弹窗文案（建议）

### 网格变更确认弹窗

- 标题：`开始新游戏`
- 内容：`你选择了 6x6 网格。确认后将开始一局新游戏，当前布局不会延续。`
- 按钮：
  - 次按钮：`取消`
  - 主按钮：`确认开始`

### 无可继续对局提示

- 文案：`暂无可继续的游戏，请先开始新游戏。`

## 7.2 交互优先级

1. 防误操作优先于快捷操作。
2. 任何会影响局面的行为都必须明确告知后执行。

## 8. 非功能需求

1. 页面切换响应 < 200ms（常规设备）。
2. 存档写入失败需可恢复，不得导致闪退。
3. 历史记录列表 100 条内滚动流畅。
4. 关键流程（继续游戏、确认开新局）需要可测试。
5. 生成概率与胜利判定规则必须可单元测试验证。

## 9. 测试用例（核心）

## 9.1 首页与继续游戏

1. 首次安装：进入首页，继续游戏置灰。
2. 完成若干步后重进：继续游戏可点击，恢复成功。
3. 清空存档后：继续游戏再次置灰。

## 9.2 网格修改与确认

1. 首页新游戏选择 5x5 -> 弹出确认 -> 取消：不进入新局。
2. 首页新游戏选择 5x5 -> 确认：进入 5x5 新局。
3. 对局中尝试改网格：不可直接改布局（入口不存在或禁用）。

## 9.3 历史记录

1. 完成 3 局后，历史页按时间倒序展示。
2. 超过保留上限后，最旧记录被淘汰。
3. 进行中记录状态更新正确。

## 9.4 生成概率与动态目标

1. 统计 1000 次新方块生成：`2` 比例约 90%，`4` 比例约 10%。
2. 4x4 对局目标应为 2048，合成 2048 时触发胜利。
3. 5x5 对局目标应为 4096，合成 2048 不触发胜利，合成 4096 触发胜利。
4. 3x3 对局目标应为 256，合成 256 时触发胜利。
5. 历史记录中 `targetTile` 与该局网格匹配，`isTargetReached` 记录正确。

## 10. 里程碑建议

1. M1（基础结构）：首页 + 路由 + 新游戏入口 + 状态管理拆分。
2. M2（存档能力）：继续游戏快照保存/恢复 + 容错处理。
3. M3（历史记录）：记录模型、列表页、详情展示、上限策略。
4. M4（核心规则）：新方块概率生成 + 动态胜利目标 + UI 展示改造。
5. M5（交互打磨）：网格确认弹窗、防重复触发、文案细化。
6. M6（测试验收）：核心流程测试 + 规则分布测试 + 回归验证 + 文档更新。
